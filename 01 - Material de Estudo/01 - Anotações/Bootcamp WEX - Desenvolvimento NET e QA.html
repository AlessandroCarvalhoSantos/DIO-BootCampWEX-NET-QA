          • O que é versionamento de código
            • O versionamento de código é o processo de gerenciar e controlar as diferentes versões de um software ao longo do tempo. Ele é essencial para facilitar o desenvolvimento colaborativo, rastrear mudanças, solucionar problemas e manter um histórico das modificações realizadas em um projeto de software.
            • Sistemas de Controle de Versão
              • Controlam as versões de um arquivo ao longo do tempo.
              • Registra o histórico de atualizações de um arquivo;
              • Gerencia quais foram as alterações, a data, autor, etc.;
              • Organização, controle e segurança.
            • Tipos de sistemas de controle de versão
              • Dentro os sistemas de controle de versão (VCS), temos:
                • VCS Centralizado (CVCS)
                  • Servidor centralizado, edição dos arquivos de forma remota
                    • Desvantagens de ficar fora do ar, fica sem mexer
                  • EX: CVS E Subversion
                • VCS Distribuído (DVCS)
                  • Servidor centralizado + repositório localizados (Ecossistema descentralizado)
                  • Clona é como backup, clona tudo desde o histórico até arquivos antigos e informações de todos do projeto
                  • Possibilita trabalhar sem a rede e trabalho flexível
                  • Git e Mercurial
            • Existem várias ferramentas e sistemas de controle de versão disponíveis, sendo os mais populares:
              1. Git: É um sistema de controle de versão distribuído amplamente utilizado. Permite que vários desenvolvedores trabalhem em paralelo em um mesmo projeto e mantém um histórico detalhado de todas as mudanças.
              1. Subversion (SVN): É um sistema de controle de versão centralizado. Embora não seja tão flexível quanto o Git em termos de trabalho colaborativo, ainda é usado em alguns projetos.
              1. Mercurial: Assim como o Git, é um sistema de controle de versão distribuído. Embora menos popular do que o Git, oferece funcionalidades semelhantes.
          • O que é GIT
            • O Git é um sistema de controle de versão distribuído amplamente utilizado para o desenvolvimento de software. Foi criado por Linus Torvalds, o mesmo criador do kernel do Linux, e é conhecido pela sua eficiência, velocidade e flexibilidade.
            • Gratuito e open source
            • Sistema de ramificações (branch) e fusões (merge)
            • Leve e rápido
            • Site oficial do git
            • Breve histórico do git
              • 2002 → O projeto do núcleo (kernel) do Linux, que é open source, começa a utilizar o BitKeeper, um DVCS proprietário;
              • 2005 → Após conflitos com a comunidade, o BitKeeper rescinde a licença gratuita. O que leva a linux Torvalds, o criador do Linux, e sua equipe desenvolverem sua própria ferramenta, o Git.
            • Fluxo Básico do GIT
              • Quando queremos clonar um repositório damos o seguinte comando
              • git clone → Clona um repositório git existente para um novo diretório (pasta) local.
              • git commit → Grava alterações no repositório
              • git pull → “Puxa” as alterações do repositório remoto para o local (busca e mescla)
              • git push → Empurra as alterações do repositório local para o remoto
              • Fluxo
                • Clonar → puxar ←→ comitar ←→ subir
          • O que é GitHub
            • O GitHub é uma plataforma de hospedagem de código fonte e colaboração para desenvolvimento de software. Ele oferece uma série de ferramentas e recursos que facilitam o gerenciamento de projetos, o controle de versões e a colaboração entre equipes de desenvolvimento.
            • Comunidade ativa
            • Utilizado mundialmente
            • Breve Histórico do GitHub
              • 2008 → Desenvolvido por Chris Wanstrath, J. Hyett, Tom Preston-Werner e Scott Chacon.
              • 2018 → Vítima de um dos maiores ataques de DDoS (ataque distribuído de negação de serviço). Comprado pela Microsoft Corporation por US $ 7,5 bilhões
            • Git e GitHub são coisa diferentes
            • Criação da conta normalmente
            • Adicionar autenticação de dois fatores

          • Configurando o git
            • Quando abrimos a o git bash (terminal), temos o nome do usuário, o git e pasta atual
            • Podemos alterar o tema, só clicar botão direito no terminal e ir options e depois em temas
            • git config → mostra as configurações do git além de pode modificar variáveis do git
              • As mais importantes são a global (Configurações do usuário), system (do sistema) e local (Configurações locais)
            • Ctrl + L → limpa o terminal
            • Definindo o nome de usuário de forma global
              • O nome fica salvo nos commits
              • Se alterar o nome, ele não altera nos commits antigos
            git config --global user.name "NomeUser"
            • Definindo o email de usuário de forma global
            git config --global user.email email@xxxx.xx
            • Caso queira ver as config
              • git config user.name
              • git config user.email
            • Configurando o nome da branch padrão
              • Exibe o nome da branch padrão quando criada
              git config init.defaultBranch
              • Alterando o valor
              git config --global init.defaultBranch main
              • Podemos alteras inúmera outras coisa, podemos ler mais na documentação
              • Informações sobre configurações iniciais
              • Mostra todas as configs (global, system, local)
                • A local só funciona quando tiver algum repositório local 
                git config --global --list
git config --local --list
git config --system --list
          • Autenticação via Token
            • Desde 2021, o github deixou de usar senhas, agora ele usa tokens
            • O uso de token tem vantagens como
              • Único – os tokens são específicos do GitHub e podem ser gerados por uso ou por dispositivo
              • Revogável – os tokens podem ser revogados individualmente a qualquer momento, sem a necessidade de atualizar credenciais não afetadas
              • Limitado – os tokens podem ter um escopo restrito para permitir apenas o acesso necessário para o caso de uso
              • Aleatório – os tokens não estão sujeitos aos tipos de dicionário ou tentativas de força bruta que as senhas mais simples que você precisa lembrar ou inserir regularmente podem estar
            • Se tentar usar senha e login, ele dar um erro e pede para autenticar com o token
            • Pelas configurações podem acessar a parte de criar tokens, e configurar ele com um nome e o tempo de expiração, além das permissões, uma vez criado ele não fica mais visível na página, a ideia e ser usado uma vez e ficar ciclando ele.
            • Podemos salvar o token na maquina, para autenticações futuras.
            • Comando para adicionar o token globalmente
              git config --global credential.helper store
            • Depois só executar o comando novamente e seu está salvo
            • Mais informações
            • Ver qual tipo está salvando
              git config --global credential.helper
            • Ver qual local ele salvou, 
              git config --global --show-origin credential.helper
            • Dentro da pasta tem duas configurações, a geral e a credencial salva
            • Podemos dar o comando cat + arquivo para mostra o conteúdo do arquivo no terminal
              cat .gitconfig
            • Tenha cuidado com credenciais salva no windows do git pode dar errado dependendo do token
            • o ~ do terminal pode significar um alias de um caminho.
          • Autenticação via Chave SSH
            • Tópicos de ajuda
            • Esse protocolo trabalhar com um par de chaves, um pública e uma privada, a chave pública fica no git hub e a privada fica no nosso computador
            • Saber se existem chaves na maquina atual
              ls -al ~/.ssh
            • Por padrão eles ficam armazenado no home do usuário
            • O comando ls é para listar arquivos e o al é para exibir e listar junto com arquivos ocultos
            • O $ em um terminal é o início de um novo comando
            • Gerar nova chave ssh
              ssh-keygen -t ed25519 -C "your_email@example.com"
              • O ed25519 é um tipo de algoritmo de criptografia, -c é comentário e na frente um identificador da chave (como comentário)
              • Depois ele pede um local para salvar, deixamos padrão
              • E depois uma senha
              • E ele gera um chave
              • Agora vamos armazena no ssh-agent para ficar de maneira segura
              • Inicializando o serviço
                eval "$(ssh-agent -s)"
              • Depois ele uma mensagem de pid e um código de serviço se tudo der certo
              • Agora adicionamos a chave ao serviço
                ssh-add ~/.ssh/id_ed25519
              • Agora é adicionar a chave ao github
              • Depois vamos em settings e depois SSH, e adicionar, damos um título e escolhemos o tipo de autenticação autentication key, a outra é para a assinatura.
              • Agora vai no git bash e copie a chave pública
                cat id_ed25519.pub
              • A depois colocamos no github a chave
        • Primeiros Passo com git e GitHub
          • Criando e Clonando Repositórios
            • Criando e Clonando Repositórios
              • Existem duas formas de obter um repositório Git na sua máquina:
                • Transformando um diretório local que não está sob controle de versão, num repositório Git, utilizando o git init;
                • Clonando um repositório Git existente.
            • Utilizando o comando git init para poder inicializar um repositório git, no terminal aparece o nome da branch, e ele criar um pasta oculta .git na pasta em que você solicitou, e se entramos nela com terminal aparece (GIT_DIR!) , significa que é um diretório git
            • Essa pasta serve para gerenciar nosso repositório
            • Git clone - clona um repositório remoto
              • Podemos passar mais um parâmetro para nomear a pasta inicial
              git clone url.git nome_repo
            • Repare que ao inicializar com git init ele não tem url remota , ao clonar sim
            • Podemos ver a conexão remota com o seguinte comando
              git remote -v
            • Para adicionar usamos o seguinte comando
              git remote add origin url
            • o origin é alias para a url, depois só utilizando o alias para ficar mais fácil, matemos o alias como origin por padrão mesmo
            • Caso queira clonar uma branch faremos da seguinte forma
              git clone URL --branch nomeBranch
            • Caso queria somente uma branch (Caso não passe a branch, ele clona só a principal)
              git clone URL --branch nomeBranch --single-branch
git clone URL --branch --single-branch  //Clona a principal somente
            • Podemos criar os repositório pelo git hub, definimos o nome uma descrição, e se é público ou privado, podemos adicionar um readme (Arquivos para descreve nosso projeto) e o .gitIgnore (ignorar arquivos, aqueles não desejamos enviar para repositório remoto) e no final temos a configuração de licença.
            • Se a gente criar o readme ele vai apara uma página de ajuda para configurar o projeto. Mostra como inicializar, importa manualmente ou via terminal.
            • Também temos a área de settings para gerenciar nosso repositório no github.
          • Salvando alterações no repositório local
            • Git status → Mostra nossa área de trabalho ou os index de commit, além de mostrar os estados do arquivo
            • Comando touch, esse comando é de terminal e ele cria um arquivo pedido
              touch nomeArquivo.extensão
            • README.md → É um arquivo em uma linguagem especifica de marcação, que é utilizada para estila e estruturar informações do nosso projeto, geralmente colocamos ele na área inicial do projeto, o github possui um editor especial para e o consegue renderizar. A extensão .m é de markdown.
              • Dar pra botar bastante coisa no markdown, gráficos interativos e diagrama interativos.
              • Edição online de markdown
            • Adicionando arquivos em stage para fazer commit
              git add arquivo.extenssao //Adiciona arquivo para stage
git add --all // Adicionar todos os arquivos modificados para stage
git add . // Adicionar todos os arquivos modificados para stage
            • Depois do stage faremos o commit (-m é de mensagem e a mensagem é uma descrição do commit)
              git commit -m "Comentário do commit"
            • git log → Mostra os logs dos commit com autores e descrições e data
              git log
            • O git add não adiciona ou reconhece pasta vazias
            • Utilizando .ignore
              • Podemos criar um arquivo .gitignore, nele podemos adicionar pastas e arquivo que não vão ser adicionados
              README.md
Arquivos/
            • Em alguns repositório podemos ter .gitkeep que é um arquivo para preencher pasta vazias. Assim o git reconhece a pasta para subir pro github
          • Desfazendo alterações no repositório local
            • Como remover o .git a força (Isso é comando via terminal), usado mais para quando for um git init errado
              rm -rf .git
            • Restaurar arquivos ou pastas com arquivos deletado ou modificados
              git restore
            • Como alterar a mensagem do ultimo commit
              git commit --amend -m "Adiciona ignore e diretorios aulas e resumos"
            • Outra forma é editar pelo editor
              git commit --amend
            • Resetando commits
              git reset --soft
git reset --mixer
git reset --hard
              • git reset soft + hash pega os arquivos do commit solicitado e volta para o stage
                • Podemos usar o git log para pegar o hash do commit
                git reset --soft "52125asdsa52d1sa25d1as2d"
              • git reset mixed + hash pega os arquivos e volta ao original modificados, não em stage igual ao soft
                git reset --mixed "52125asdsa52d1sa25d1as2d"
              • git reset hard + hash reserta o commit e deletas os arquivos feitos
                git reset --hard "52125asdsa52d1sa25d1as2d"
            • Podemos ver o histórico de alterações
              git reflog
            • Lembrando que isso deve ser feito localmente é melhor
              • Se você fizer o commit errado, é melhor fazer um outro commit corrigindo do que desfazer, pois tem histórico e pode gerar problemas maiores.
            • Podemos usar o reset no stage também,
              • Ele tira o arquivo do stage e manda par área de trabalho normal
                git reset nomeArquivo
              • Outra opção é usar o git restore mas passando da seguinte forma
                git restore --staged nomeArquivo.extensao
            • Tomar cuidar ao botar algo em stage, alterar de novo o arquivo e dar um restore ou reset, pois ele sobrepõe os arquivos da área de trabalho
          • Enviando e Baixando alterações com o repositório remoto
            • Primeiro temo que conectar o repositório remoto ao local com o seguinte código
              git remote add origin URL
            • Depois podemos botar a git branch principal forçando ela para o nome que você quiser
              git branch -M main
            • Depois de criado o commit podemos subir com o seguinte comando
              git push -u origin main
            • O -u configura a up stream para subir, origin é alias da URL e depois na branch main
              • Lá no github podemos ver os commits
            • O github temo inúmero atalhos
              • Podemos apertar ? para ver os atalhos
              • Ao apertar . abrirá um editor de código do visual studio
            • Podemos usar o visual studio para subir commits pela interface, tanto na versão web ou instalada
            • Para baixar as atualizações usado o comando git pull
              git pull
